# FinOps for AI: The 4 KPIs Every Engineering Team Must Track

The AI gold rush is burning through cloud budgets faster than most finance teams can track. I've seen companies spend $200K/month on LLM inference without knowing their cost per customer interaction. Others run GPU clusters at 30% utilization while paying for 100%.

If you're shipping AI features, you need AI-specific FinOps metrics. Not tomorrow—today. Here are the four KPIs that separate optimized AI operations from expensive chaos.

## Why Traditional Cloud Metrics Fail for AI

Traditional FinOps relies on metrics like cost per service, cost per environment, or cost per team. These work for standard compute but break down for AI workloads because:

- **Token-based pricing** creates variable costs per request (unlike fixed EC2 pricing)
- **GPU resources** are 10-50x more expensive than CPU but it is harder to measure their utilization
- **Model inference** has complex cost drivers (model size, context length, caching)
- **Latency requirements** force tradeoffs between cost and user experience

You need metrics that capture the economics of intelligence, not just infrastructure.

## KPI #1: Cost Per Token

**What it measures:** Your effective price for processing input and output tokens across all LLM providers.

**Why it matters:** Token pricing varies wildly—GPT-4 Turbo costs $0.01/1K input tokens, Claude 3.5 Sonnet costs $0.003/1K, and open-source models on your own infrastructure might cost $0.0001/1K. But the sticker price isn't your true cost.

### How to Calculate

```python
# Basic formula
cost_per_token = total_llm_spend / total_tokens_processed

# More accurate: separate input/output
input_cost_per_token = llm_spend / (input_tokens + (output_tokens * output_price_ratio))
output_cost_per_token = input_cost_per_token * output_price_ratio

# Example with real data
monthly_spend = 15000  # USD
input_tokens = 500_000_000
output_tokens = 100_000_000
provider_input_rate = 0.003 / 1000  # per token
provider_output_rate = 0.015 / 1000  # per token

# Effective rate including infrastructure overhead
actual_cost_per_1k_input = (monthly_spend /
    (input_tokens + output_tokens * (provider_output_rate/provider_input_rate))) * 1000
```

### What Good Looks Like

| Scenario | Target Cost/1K Tokens | Notes |
|----------|----------------------|-------|
| Chatbot (GPT-4) | $0.01 - $0.03 | Includes input + output blended |
| Summarization (Claude) | $0.005 - $0.015 | Lower output ratio |
| Code completion | $0.001 - $0.005 | Cached prompts help |
| Self-hosted open source | $0.0002 - $0.002 | GPU amortization included |

### Optimization Levers

**Prompt caching:** Can reduce costs 50-90% for repeated context
```python
# Without caching
cost_per_request = (system_prompt_tokens + user_prompt_tokens + output_tokens) * price_per_token

# With caching (90% of system prompt cached)
cached_tokens = system_prompt_tokens * 0.9
cache_write_cost = cached_tokens * cache_write_multiplier  # one-time
cache_read_cost = cached_tokens * cache_read_multiplier   # 90% cheaper
cost_per_request = cache_read_cost + (user_prompt_tokens + output_tokens) * price_per_token
```

**Model routing:** Route simple queries to cheaper models
- Use GPT-3.5/Claude Haiku for 70% of queries → 80% cost reduction on those requests
- Reserve GPT-4/Claude Opus for complex tasks

**Batch processing:** 50% discount on OpenAI batch API for non-realtime workloads

## KPI #2: Cost Per Inference

**What it measures:** Total cost to serve one model prediction, including compute, storage, orchestration, and overhead.

**Why it matters:** While cost/token works for LLMs, many AI workloads use non-transformer models (embeddings, classifiers, computer vision). Cost per inference normalizes across model types and captures total operational cost.

### How to Calculate

```python
# Formula
cost_per_inference = (
    model_compute_cost +
    infrastructure_overhead +
    data_transfer_cost +
    storage_cost
) / total_inferences

# Real example: Self-hosted embedding model
gpu_cost_per_hour = 1.20  # g5.xlarge spot
cpu_overhead = 0.15  # Load balancer, monitoring
inferences_per_hour = 50000
data_egress_per_1k_inferences = 0.02  # USD

cost_per_inference = (
    (gpu_cost_per_hour + cpu_overhead) / inferences_per_hour +
    data_egress_per_1k_inferences / 1000
)
# = $0.000027 per inference

# Compare to API: OpenAI embeddings
api_cost_per_inference = 0.0001 / 1000  # $0.0001 per 1K tokens, ~1 token/inference
# = $0.0000001 per inference
```

### What Good Looks Like

| Model Type | Target Cost/Inference | Break-even vs API |
|------------|----------------------|-------------------|
| Embeddings (self-hosted) | $0.00001 - $0.00005 | >10M inferences/month |
| Image classification | $0.0001 - $0.001 | Depends on batch size |
| LLM inference (self-hosted) | $0.001 - $0.05 | >1M requests/month |
| Real-time speech-to-text | $0.002 - $0.01 | High utilization needed |

### The Build-vs-Buy Calculation

```python
# When does self-hosting beat API pricing?
api_cost_per_inference = 0.0001
self_hosted_fixed_cost_monthly = 2000  # GPU + infra
self_hosted_variable_cost = 0.00001  # per inference

# Break-even volume
break_even = self_hosted_fixed_cost_monthly / (api_cost_per_inference - self_hosted_variable_cost)
# = 2000 / (0.0001 - 0.00001) = 22.2M inferences/month
```

**Key insight:** Self-hosting only wins at scale. Below break-even, every inference costs you money.

## KPI #3: GPU Utilization Efficiency

**What it measures:** Percentage of GPU compute actually used for inference vs idle time.

**Why it matters:** A $10K/month A100 GPU at 30% utilization is a $7K/month waste. Unlike CPUs that can be right-sized easily, GPUs are chunky resources—you can't rent 30% of a GPU (though multi-instance GPUs help).

### How to Calculate

```python
# Using NVIDIA DCGM or cloud provider metrics
gpu_utilization_pct = (actual_gpu_compute_time / total_available_time) * 100

# Memory utilization (also critical)
gpu_memory_utilization_pct = (used_vram / total_vram) * 100

# Effective cost per utilized hour
effective_cost = actual_cost / (gpu_utilization_pct / 100)

# Example
gpu_cost_per_hour = 4.00  # p4d.24xlarge on-demand
average_utilization = 45
effective_cost_per_utilized_hour = 4.00 / 0.45  # = $8.89
```

### What Good Looks Like

| Workload Type | Target Utilization | Notes |
|---------------|-------------------|-------|
| Production inference (realtime) | 60-80% | Leave headroom for spikes |
| Batch training | 85-95% | No realtime requirements |
| Development/experimentation | 40-60% | Expected due to iteration |
| Idle instances | 0-10% | Should be auto-stopped |

### Optimization Strategies

**1. Autoscaling with smart thresholds**
```yaml
# Karpenter/GKE autoscaler example
apiVersion: karpenter.sh/v1alpha5
kind: Provisioner
metadata:
  name: gpu-inference
spec:
  requirements:
    - key: node.kubernetes.io/instance-type
      operator: In
      values: ["g5.xlarge", "g5.2xlarge"]
  limits:
    resources:
      nvidia.com/gpu: 10
  # Scale down after 5min idle
  ttlSecondsAfterEmpty: 300
```

**2. Multi-tenancy with MIG (Multi-Instance GPU)**
- Split A100 into 7 instances (1g.5gb each)
- Each tenant gets isolated GPU slice
- Utilization compounds: 7 workloads at 30% each = 210% effective utilization

**3. Spot instances for fault-tolerant workloads**
- 60-70% discount vs on-demand
- Training workloads with checkpointing
- Batch inference with retry logic

**4. Time-based scheduling**
```bash
# Stop dev GPUs outside working hours
# Savings: 16 hours/day * 5 days/week = 67% cost reduction
0 18 * * 1-5 kubectl scale deployment gpu-dev --replicas=0
0 8 * * 1-5 kubectl scale deployment gpu-dev --replicas=3
```

## KPI #4: Time to First Token (TTFT)

**What it measures:** Latency from request submission to first token streamed back to user.

**Why it matters:** This is where cost meets user experience. Optimizing for pure cost (bigger batches, slower models, aggressive caching) can destroy latency. TTFT lets you set cost/latency guardrails.

### How to Calculate

```python
import time

start = time.time()
response = openai.ChatCompletion.create(
    model="gpt-4-turbo",
    messages=[{"role": "user", "content": prompt}],
    stream=True
)

first_token_time = None
for chunk in response:
    if first_token_time is None:
        first_token_time = time.time()
        ttft = first_token_time - start
        break

# Track percentiles
ttft_p50 = 0.450  # seconds
ttft_p95 = 1.200
ttft_p99 = 2.500
```

### What Good Looks Like

| Use Case | Target TTFT P95 | Cost/Latency Tradeoff |
|----------|----------------|----------------------|
| Chatbot | <800ms | Users notice >1s delay |
| Code completion | <200ms | Needs to feel instant |
| Document summarization | <2s | Batch-friendly, cost-optimized |
| Background analysis | <10s | Pure cost optimization |

### The Cost/Latency Frontier

```python
# Scenario comparison
scenarios = [
    {"name": "GPT-4 Turbo", "ttft_p95": 0.6, "cost_per_1k": 0.03},
    {"name": "GPT-3.5 Turbo", "ttft_p95": 0.3, "cost_per_1k": 0.002},
    {"name": "Self-hosted Llama", "ttft_p95": 0.8, "cost_per_1k": 0.0005},
    {"name": "Batch API", "ttft_p95": 300, "cost_per_1k": 0.015},  # 5min latency
]

# Pick based on requirements
def choose_model(max_latency_sec, volume_per_month):
    viable = [s for s in scenarios if s["ttft_p95"] <= max_latency_sec]
    monthly_costs = [(s["name"], s["cost_per_1k"] * volume_per_month / 1000)
                     for s in viable]
    return min(monthly_costs, key=lambda x: x[1])
```

### Optimization Without Sacrificing Latency

**1. Model size selection**
- Llama 3.1 8B: TTFT ~200ms, lower quality
- Llama 3.1 70B: TTFT ~600ms, better quality
- Choose based on use case

**2. Speculative decoding**
- Small model drafts tokens, large model verifies
- 2-3x speedup on throughput, minimal quality loss

**3. KV cache optimization**
- Keep frequent prompt prefixes in memory
- Reduces TTFT by 40-60% for repeated contexts

**4. Geographic distribution**
- Deploy models closer to users
- Each 1000 miles ≈ 20-30ms added latency
